# 开发检查清单

在提交代码前，请确保完成以下检查。

## ✅ 代码质量检查

### TypeScript

- [ ] 所有类型定义完整，没有 `any` 类型
- [ ] 类型检查通过：`pnpm typecheck`
- [ ] 没有类型错误或警告
- [ ] 使用了正确的类型工具（如 `DeepReadonly`、`Optional` 等）

### ESLint

- [ ] ESLint 检查通过：`pnpm lint`
- [ ] 没有 ESLint 错误
- [ ] 代码已自动格式化（通过 lint-staged）

### 代码组织

- [ ] 代码放在正确的位置（应用层/业务域/UI/共享）
- [ ] 没有违反模块边界规则
- [ ] 使用正确的路径别名导入
- [ ] 文件命名符合规范

## 📝 代码规范检查

### 导入规范

- [ ] 使用路径别名而非相对路径（超过 2 层）
- [ ] 导入顺序正确（外部库 → 内部库 → 相对路径）
- [ ] 没有未使用的导入

### 命名规范

- [ ] 组件使用 PascalCase
- [ ] 函数和变量使用 camelCase
- [ ] 常量使用 UPPER_SNAKE_CASE
- [ ] 类型和接口使用 PascalCase

### 代码风格

- [ ] 代码已格式化（Prettier）
- [ ] 没有过长的函数（建议 < 50 行）
- [ ] 没有过深的嵌套（建议 < 4 层）
- [ ] 注释清晰、有意义

## 🧪 功能检查

### 功能完整性

- [ ] 功能按需求实现
- [ ] 边界情况已处理
- [ ] 错误处理完善
- [ ] 用户体验良好

### 测试

- [ ] 单元测试通过（如有）
- [ ] 手动测试通过
- [ ] 没有明显的性能问题

## 📚 文档检查

### 代码文档

- [ ] 复杂逻辑有注释说明
- [ ] 公共 API 有 JSDoc 注释
- [ ] 类型定义有说明

### 提交信息

- [ ] 提交信息遵循 Conventional Commits 规范
- [ ] 提交信息清晰描述变更内容
- [ ] 相关 Issue 已关联（如有）

## 🔍 提交前检查

运行以下命令确保一切正常：

```bash
# 1. 类型检查
pnpm typecheck

# 2. Lint 检查
pnpm lint

# 3. 构建检查（可选）
pnpm build

# 4. 测试（如有）
pnpm test
```

## 🚫 常见错误

### ❌ 不要做

- 不要使用 `any` 类型
- 不要使用 `@ts-ignore` 或 `@ts-expect-error`（除非有充分理由）
- 不要违反模块边界规则
- 不要使用超过 2 层的相对路径
- 不要在组件中直接调用 API（使用应用层包装）

### ✅ 应该做

- 使用类型守卫进行运行时检查
- 使用统一的错误处理
- 遵循代码组织规范
- 编写清晰的提交信息
- 保持代码简洁、可读

## 📋 快速检查命令

```bash
# 一键检查所有
pnpm check

# 或分步检查
pnpm typecheck && pnpm lint
```
