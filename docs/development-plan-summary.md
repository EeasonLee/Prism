# 开发计划快速参考

> 本文档是 `development-plan.md` 的快速参考版本，包含关键信息和时间线。

## 🎯 核心目标

1. ✅ 写代码时就能发现 TS / Lint 错误
2. ✅ 新增业务不会影响旧模块
3. ✅ 任何人都知道代码该放哪
4. ✅ 规范不是靠自觉，而是靠工具
5. ✅ 你敢让别人接手这个项目

## 📅 时间线概览

| 阶段        | 时间     | 关键任务                | 状态      |
| ----------- | -------- | ----------------------- | --------- |
| **Phase 1** | Week 1-2 | 目录结构重构 + 基础规范 | ⏳ 待开始 |
| **Phase 2** | Week 3-6 | 核心能力建设            | ⏳ 待开始 |
| **Phase 3** | 持续     | 优化与完善              | ⏳ 待开始 |

## 🚀 Phase 1: 基础架构（Week 1-2）

### Week 1: 目录结构重构

**Day 1-2: 准备阶段**

- 制定目标结构文档
- 建立路径别名体系
- 创建 libs 目录结构

**Day 3-5: 代码迁移**

- 迁移共享代码到 `libs/shared/`
- 迁移业务代码到 `libs/[domain]/`
- 创建 import 路径迁移工具

**Day 6-7: 路径迁移**

- 执行 import 路径迁移
- 验证和测试

### Week 2: 基础规范强化

**Day 1-2: 类型体系**

- 建立 TS 类型体系
- 创建通用响应类型

**Day 3-4: 规范强化**

- 强化 ESLint 规则（error 级别）
- 强化 Git Hooks 和 CI

**Day 5-7: 文档完善**

- 完善架构文档
- 创建开发指南

## 🔧 Phase 2: 核心能力（Week 3-6）

### Week 3: 请求/错误/权限统一收口

- API 类型体系完善
- 错误处理统一
- 权限系统设计
- 日志系统增强

### Week 4: 数据 & 状态架构

- 状态管理方案确定
- 数据获取层设计
- 状态同步机制
- 测试与验证

### Week 5: UI 与业务逻辑解耦

- 组件分层设计
- 业务逻辑提取
- 组件库建立
- 组件文档

### Week 6: Strapi 集成

- Strapi SDK 封装
- 内容类型映射
- 媒体处理
- 测试与验证

## 📋 关键检查点

### Phase 1 完成标准

- [ ] 目录结构符合规范
- [ ] 所有 import 使用路径别名
- [ ] TS 类型体系建立
- [ ] ESLint 规则为 error 级别
- [ ] 架构文档完整

### Phase 2 完成标准

- [ ] 请求/错误/权限统一收口
- [ ] 数据架构清晰
- [ ] UI 与业务逻辑解耦
- [ ] Strapi 集成完成
